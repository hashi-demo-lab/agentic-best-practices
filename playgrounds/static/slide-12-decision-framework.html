Decision Framework

When to Use Subagents vs Alternatives
New Task
Interactive?
Reusable?
Parallel?
Choose Model
Subagents Isolation
Verbose output, tool restrictions, self-contained work, parallel investigation
Task produces verbose output that would bloat parent context
Need to restrict tool access for safety or efficiency
Work is self-contained with clear inputs and outputs
Parallel investigation is possible across independent areas
Main Conversation Interactive
Frequent back-and-forth, shared context, quick changes, latency-sensitive
Task needs frequent back-and-forth with the user
Multiple phases share context and build on each other
Making quick, focused changes to a few files
Response latency matters -- subagent round-trip adds overhead
Skills Reusable
Reusable prompts/workflows, reference knowledge, invoked on trigger
Need reusable prompts or step-by-step workflows
Reference knowledge like conventions, patterns, style guides
Invoked by user command or Claude detecting a matching pattern
Domain knowledge that multiple agents or sessions can share
Headless Mode Automation
Process-level parallelism, batch processing, CI/CD pipelines
True process-level parallelism needed (not just subagent concurrency)
Batch processing like file migrations across many modules
CI/CD pipeline integration -- agent runs unattended
Tasks are fully independent with no shared state needed

Key Principle

Match the execution model to the task characteristics. Subagents for isolation and parallelism, main conversation for interactivity, skills for reusable knowledge, headless for batch automation.
