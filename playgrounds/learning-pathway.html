Agentic Best Practices

Prerequisites

Required software and environment variables

Docker Desktop

Required for running the devcontainer. Download Docker Desktop from docker.com and ensure it is running before proceeding.

VS Code

Recommended IDE with devcontainer support. Install the Dev Containers extension from the VS Code marketplace.

Environment Variables

Two tokens are required to authenticate with GitHub and HCP Terraform:

GITHUB_TOKEN    # GitHub PAT with repo permissions
TEAM_TFE_TOKEN  # HCP Terraform Team Token (must be a Team API Token, not user/org token)

Important: TEAM_TFE_TOKEN

Must be a Team API Token — not a user token or organization token. Create at:

HCP Terraform > Settings > Teams > [Your Team] > Team API Token

HCP Terraform Setup

Configure HCP Terraform with an isolated environment for testing

1. Create a Dedicated Project

Navigate to Projects in HCP Terraform and create a new project (e.g., sandbox). This isolates test workspaces from production.

2. Create a Dedicated Team

Go to Settings > Teams. Create a team and assign it to your project with the following permissions:

Read, Create Workspaces, Delete Workspaces,
Read Variables, Read State, Write State,
Lock/Unlock Workspaces

3. Generate Team API Token

In Settings > Teams > [Your Team], click "Create a team token". Save this value as TEAM_TFE_TOKEN.

4. Configure Credential Inheritance

Create a Variable Set with AWS credentials and attach it to your dedicated project. Workspaces inherit credentials automatically.

5. AWS Credentials

Prefer Dynamic Provider Credentials (OIDC federation) or a Variable Set with:

AWS_ACCESS_KEY_ID
AWS_SECRET_ACCESS_KEY
AWS_REGION

6. Shell Configuration

Add your tokens to your shell profile so they are available in every session:

# Add to ~/.bashrc or ~/.zshrc
export GITHUB_TOKEN="ghp_your_token_here"
export TEAM_TFE_TOKEN="your_terraform_team_token_here"

# Reload
source ~/.bashrc  # or source ~/.zshrc

Skills: Domain Knowledge for Agents

SKILL.MD files package reusable expertise that the agent selects by matching name and description

Name & Description Matching

Agents discover skills by scanning name and description fields. The orchestrator matches the current task context against available skills and loads only those that are relevant.

concept-skills
Skills are not pre-loaded — the agent decides which skills to pull in based on the task at hand.

Progressive Disclosure

Metadata is loaded first (name, description, triggers). Full skill content is only loaded on-demand when the agent decides to invoke the skill, preserving context window space.

This two-phase loading keeps the agent's context lean until a skill is actually needed.

Version-Controlled

Skills live in .claude/skills/ and are version-controlled alongside the codebase. Changes to skills are reviewed via standard PR workflows.

tf-module-plan
terraform-style-guide
tf-research
Stored as markdown files under .claude/skills/ — portable, diffable, and composable.

User-Invocable Slash Commands

Skills can be triggered directly by users via slash commands (e.g. /tf-module-plan). This gives humans the same entry points that agents use programmatically.

Slash commands bridge the gap between interactive use and autonomous execution.

Subagents: Custom Specialists

Markdown files that define specialized AI assistants with scoped tools, models, and permissions

Scoped Capabilities

Each subagent gets only the tools it needs. A research agent gets MCP read access; a developer agent gets file-write permissions. No agent has more capability than its task requires.

concept-subagents
Blast radius containment — a subagent failure doesn't crash the parent workflow.

Context Extension

Each subagent runs in an isolated context window. This effectively multiplies the available context across the workflow — the orchestrator stays lean while specialists go deep.

Isolated windows prevent context pollution between unrelated tasks.

Built-in Types

Three built-in subagent types cover common patterns: Explore (read-only investigation), Plan (analysis and design), and General-purpose (flexible execution).

Built-in types provide sensible defaults; custom agents extend them for domain-specific work.

Custom Specialists

Define custom subagents as markdown files with specific tool permissions, model selection, and behavioral instructions tailored to each role in the workflow.

tf-module-research
tf-module-developer
tf-module-validator
Custom agents are defined in .claude/agents/ — each tailored to a specific workflow role.

Context Engineering

Why multi-agent architectures require deliberate context isolation

Single-Agent Limits

A single agent tackling a complex task fills its context window with research, code, test output, and validation results. As context grows, quality degrades — the agent loses track of earlier decisions and makes contradictory choices.

concept-context-engineering
Context saturation is the primary failure mode of monolithic agent architectures.

Multi-Agent Solution

Each specialist agent gets a clean, focused context window. A research agent only sees research materials. A developer only sees design specs and code. No cross-contamination between concerns.

Clean context windows mean each agent operates at peak quality for its specific task.

Orchestrator Stays Lean

The orchestrator distributes tasks to specialists and synthesizes their results. It never loads full research findings or code — only summaries and status updates flow back to the coordinator.

The orchestrator's context contains workflow state, not work product.

60–75% Headroom

Target 60–75% context utilization per agent. Reserve the remaining capacity for reasoning, error recovery, and unexpected edge cases that require additional context loading.

Agents that run at 90%+ context utilization produce noticeably worse output.

Concurrent Execution: Fan-Out Research

Parallel foreground dispatch for multi-source investigation

Parallel Foreground

The orchestrator dispatches 3–4 research agents in a single message. Each agent investigates one specific question concurrently, dramatically reducing wall-clock research time.

concept-concurrency
tf-module-research×3-4
Fan-out pattern: one orchestrator message spawns multiple parallel agents.

Concise Returns

Each research agent answers exactly one question and returns a focused summary. Responses are kept under 500 tokens to prevent the orchestrator's context from bloating.

One question per agent — no scope creep in research tasks.

Disk-Based Output

Research findings are written to disk files rather than passed entirely in-memory. The orchestrator verifies file existence and reads summaries, keeping its own context lean.

Large artifacts go to disk; only metadata flows through the orchestrator.

Design Synthesis

Research findings from all parallel agents feed into the design agent. The design agent reads research files from disk and synthesizes them into a cohesive design document.

Research feeds design — the fan-out results converge into a single planning artifact.

Maximizing Throughput

Right-size models and manage context for agent performance

Right-Size Models

Match model capability to task complexity. Opus for design and research (high reasoning). Sonnet for implementation (balanced). Haiku for simple, repetitive tasks (fast and cheap).

concept-throughput
Using Opus for everything wastes tokens and time; using Haiku for design produces poor results.

Minimize Return Payload

Subagent responses should be under 500 tokens. Return structured summaries, not raw output. The orchestrator needs status and key decisions, not full implementation details.

Every token returned to the orchestrator consumes context that could be used for reasoning.

Disk-Based Artifact Passing

Large artifacts (design docs, code files, test results) are written to disk. Agents read from disk as needed rather than passing full content through the orchestrator's context.

Disk is unlimited; context windows are not. Use the filesystem as shared memory.

Scope Narrowly

Each subagent gets a single, well-defined task. A developer agent implements one checklist item, not the entire module. Narrow scope means faster completion and higher quality.

One checklist item per agent invocation — no multi-task agents.

Resume Don't Restart

Checkpoint commits after each completed item allow workflows to resume from the last successful point. If an agent fails, restart from the checkpoint — don't re-run the entire workflow.

Checkpoint commits are insurance against context loss and agent failures.

Decision Framework

When to use subagents vs direct tools vs headless mode

Main Conversation

Interactive + Focused

Use the main conversation for focused, interactive tasks. The user and agent collaborate directly with full tool access. Best for single-file edits, quick investigations, and conversational problem-solving.

concept-decision-framework
Direct interaction — no delegation overhead. Ideal when the task fits in one context.

Subagents

Interactive + Broad

Spawn subagents when the task is too broad for one context. Each subagent handles an isolated subtask with scoped tools. The orchestrator coordinates and synthesizes results.

Parallel isolated work — research, implementation, validation across multiple agents.

Skills

Autonomous + Focused

Skills inject domain knowledge for focused autonomous tasks. The agent loads a skill and executes a well-defined procedure without user interaction. Best for repeatable, codified workflows.

Codified expertise — the same procedure executes consistently every time.

Headless Mode

Autonomous + Broad

Headless mode runs complex, multi-phase workflows without user interaction. The agent orchestrates subagents, manages checkpoints, and only pauses for human approval gates.

Full autonomy with safety gates — CI/CD pipelines and batch processing.

Agentic Loop

Validation-driven cycle: write tests, write code, validate, deploy

Write Tests

The test-writer agent creates .tftest.hcl files from design.md test scenarios. Tests are written BEFORE implementation (TDD) — establishing the validation baseline.

concept-agentic-loop
tf-module-test-writer
RED phase — tests exist but code doesn't yet. Validates HCL syntax only.

Write Code

Developer agents implement one checklist item at a time from design.md Section 6. Each agent reads the design spec, writes .tf files, and runs fmt + validate + test.

tf-module-developer
GREEN phase — one item per agent, marked [x] on completion.

Validate

The validator runs the full pipeline: terraform fmt, validate, test, tflint, trivy. On FAIL, control returns to the developer for fixes. On PASS, the workflow advances to deploy.

tf-module-validator
FAIL → Write Code (fix loop)  |  PASS → Deploy (advance)

Deploy

Checkpoint commits preserve progress. A pull request is created linking to the tracking issue. HCP Terraform runs plan and apply with human approval gates at each stage.

Checkpoint commits after each item enable resume-from-failure without restarting.

Workflow Observability

Monitor agent workflows with OTEL traces and quality scoring

OTEL Telemetry Export

Agent workflows export traces, metrics, and logs via OpenTelemetry (http/protobuf). Each agent invocation is a span, enabling end-to-end workflow visualization and performance analysis.

concept-observability
Full distributed tracing — see exactly where time and tokens are spent.

6 Quality Dimensions

Output quality is scored across six weighted dimensions: Correctness, Security, Best Practices, Testing, Documentation, and Completeness. Threshold: 7.0 for production readiness.

Quantitative quality gates prevent subjective "looks good" approvals.

Structured Logging

Every log entry includes structured fields: session.id, job.name, phase, step, and status. This enables filtering, aggregation, and correlation across complex multi-agent workflows.

Structured fields make logs queryable — find all failures for a specific session instantly.

Cost & Regression Tracking

Track token usage, API costs, and quality scores over time. Detect regressions when model updates or prompt changes cause quality drops. Historical data enables optimization and budgeting.

Cost per workflow run + quality trend lines = informed optimization decisions.

Deny Permissions: Protecting Agent Guardrails

Immutable restrictions that prevent agents from weakening their own safety boundaries

Concept

Immutable Guardrails

Agents are denied write access to verification and configuration files. These files define the safety boundary — if an agent could modify them, it could effectively remove its own constraints.

concept-deny-permissions

Self-Modification Blocked

Agents cannot weaken linting rules, disable pre-commit hooks, or alter MCP configurations. This prevents a sophisticated agent from gradually eroding quality gates during long-running tasks.

Scoped Allow-List

Rather than blocking specific dangerous commands, the system uses an explicit allow-list. Only approved commands can be executed — everything else is implicitly denied.

Dangerous Ops Blocked

Operations with blast-radius implications (workspace variable creation, run confirmation) require human approval through explicit gates. The agent cannot bypass these checks.

DENY List

Agents cannot write to these files:

  • .gitignore
  • .tflint.hcl
  • .pre-commit-config.yaml
  • .mcp.json
  • create_workspace_variable
  • create_run

ALLOW List

Explicitly permitted commands:

  • terraform init / validate / fmt / plan
  • git add / commit
  • pre-commit run
  • gh issue view
  • validate-env.sh
  • create-feature.sh

Development Containers

Reproducible, isolated environments with pre-installed toolchains for consistent agent behavior

Concept
Tool

Docker-in-Docker

Enables MCP servers to run as containers within the dev container. Each MCP tool gets its own isolated runtime, preventing cross-contamination between tools.

concept-dev-containers

Consistent Toolchain

All tools are pre-installed at pinned versions: terraform, tflint, trivy, AI Agent, vault-radar, and pre-commit. No version drift between developers or CI environments.

Env Vars for Agent Config

Environment variables configure agent behavior: TFE_TOKEN, AWS_REGION, and feature flags are set in the container definition, ensuring consistent agent configuration.

Vault Radar Secret Scanning

Continuous scanning for accidentally committed secrets. Vault Radar runs as part of the pre-commit hook chain, catching credentials before they reach version control.

Team-Portable

The dev container definition is checked into the repository. Any team member (or CI runner) can spin up an identical environment in seconds. No "works on my machine" issues with agent tooling.

Agent Identity & Workload Federation

Credential-free agent operations through OIDC federation and project-scoped blast radius

Concept

Agent Never Sees Credentials

The agent operates at the HCP Terraform API layer. It never handles AWS keys, Azure secrets, or GCP service accounts directly. All cloud operations flow through the managed platform.

concept-agent-identity

OIDC Federation

Workspace-to-provider trust established via OpenID Connect. Each workspace has a federated identity that grants just enough access to the target cloud provider — no static keys stored anywhere.

Dynamic Credentials Per Run

Every Terraform run receives short-lived, dynamically generated credentials. Tokens expire after the run completes, eliminating the risk of credential leakage or reuse.

Blast Radius Contained

Workspaces are scoped to projects. An agent working on module A cannot affect the state or resources of module B. Project boundaries enforce isolation at the platform level.

Federation Flow

Agent → HCP Terraform → OIDC → Cloud Providers

The agent interacts with HCP Terraform via API — never directly with cloud provider credentials

Spec-Driven Development: 4 Phases

A structured methodology where the design document is the single source of truth

Concept

Overview

Spec-Driven Development (SDD) splits every task into 4 sequential phases. The design document produced in Phase 2 becomes the contract that all downstream agents follow. No implementation begins until the human approves the design.

concept-sdd-overview
Phase 1
Clarify

Scan requirements against the 8-category ambiguity taxonomy. Research unknowns via concurrent subagents. Flag underspecified areas and ask clarifying questions.

Output: clarified requirements + research findings
Phase 2
Design

Produce the design document — the single source of truth. Covers purpose, interface, resources, security, tests, implementation checklist, and open questions. All 7 mandatory sections must be present.

Output: design.md (the P1 planning artifact)
Human Approval Gate
Phase 3
Build + Test

TDD cycle driven by the design checklist. Test writer creates stubs first, then developers implement one checklist item at a time. Orchestrator runs trust-but-verify after each item.

Output: implementation code + passing tests
Phase 4
Validate

Quality scoring across 6 dimensions, security scanning with trivy, auto-fix for unambiguous issues (max 3 rounds). Final output: validated PR with linked tracking issue.

Output: quality score + validation report + PR

Skill Mapping

/tf-{workflow}-plan = Phases 1–2 (Clarify + Design)   •   /tf-{workflow}-implement = Phases 3–4 (Build + Validate)

Constitutions

Non-negotiable rules loaded into every agent's context — the guardrails that never change

Concept
Reference

What Are Constitutions?

Constitutions are non-negotiable rules loaded into every agent's context. They define code standards, naming conventions, security defaults, and quality thresholds. Three constitutions are tailored to each workflow type.

concept-constitutions

Module Constitution

Defines file structure, naming conventions, security defaults, and tagging requirements for Terraform modules. Ensures every module follows the same organizational pattern regardless of which agent authored it.

  • File structure: main.tf, variables.tf, outputs.tf, versions.tf
  • Naming: snake_case for resources, descriptive variable names
  • Security: encryption enabled by default, no public access
  • Tags: required standard tags on all resources
module-constitution.md

Provider Constitution

Establishes Go conventions, Plugin Framework patterns, and testing requirements for custom Terraform providers. Covers schema design, CRUD operations, and acceptance test standards.

  • Go conventions: effective Go, error handling patterns
  • Plugin Framework: schema types, validators, plan modifiers
  • Testing: acceptance tests required for every resource
  • State: proper state management and import support
provider-constitution.md

Consumer Constitution

Governs module sourcing, variable wiring, and security defaults for infrastructure consumers. Ensures teams compose modules consistently and securely.

  • Module sourcing: private registry with pinned versions
  • Variable wiring: explicit input/output mapping
  • Security: inherited security controls from modules
  • State: remote state with workspace isolation
consumer-constitution.md

Shared Principles

  • Security-first defaults — encryption, no public access, least privilege
  • Single design doc — one source of truth per task
  • Quality gates — automated validation at every phase boundary
  • Deny-by-default — explicit allow-list for all agent operations

Design Templates

Canonical markdown structures that ensure completeness, consistency, and machine-readability

Concept
Template

What Are Design Templates?

Design templates are canonical markdown structures that design agents fill in. They ensure completeness (all sections present), consistency (same format across features), and machine-readability (sections can be grep'd by downstream agents).

concept-design-templates

Module Template

7 mandatory sections:

  • 1. Purpose — What the module does and why
  • 2. Resources — AWS/cloud resources managed
  • 3. Interface — Variables, outputs, and types
  • 4. Security — Encryption, access, IAM controls
  • 5. Tests — Test scenarios and assertions
  • 6. Checklist — Implementation items
  • 7. Questions — Open items for human review
module-design-template.md

Consumer Template

6 mandatory sections:

  • 1. Purpose — Infrastructure composition goal
  • 2. Module Selection — Private registry modules to use
  • 3. Wiring — Input/output connections between modules
  • 4. Security — Inherited + additional controls
  • 5. Checklist — Implementation items
  • 6. Questions — Open items for human review
consumer-design-template.md

Provider Template

7 mandatory sections:

  • 1. Purpose — Resource/data-source being built
  • 2. Schema — Attributes, types, validators
  • 3. CRUD — Create, Read, Update, Delete operations
  • 4. State — State management and import
  • 5. Tests — Acceptance test scenarios
  • 6. Checklist — Implementation items
  • 7. Questions — Open items for human review
provider-design-template.md

Shared Sections

All three templates share two critical sections that enforce quality across every workflow:

  • Security Controls — Every template mandates an explicit security section. Agents must document encryption, access controls, and compliance requirements.
  • Implementation Checklist — A numbered list of discrete tasks. Each item becomes a unit of work for a developer subagent, enabling parallel execution.

The Design Document

The SINGLE planning artifact — human-approved before any implementation begins

Concept
Artifact

What Is the Design Document?

The design document (design.md) is the SINGLE planning artifact for every SDD task. It is the contract between the design phase and the implementation phase. Every downstream agent — test writer, developer, validator — reads from this document.

concept-design-doc
design.md

7 Mandatory Sections

  • §1 Purpose & Scope — What is being built and the boundaries of the task
  • §2 Interface Contract — Variables, outputs, types, and defaults
  • §3 Resources & Architecture — Cloud resources and their relationships
  • §4 Security Controls — Encryption, IAM, access, logging
  • §5 Test Scenarios — Test cases mapped to acceptance criteria
  • §6 Implementation Checklist — Numbered items, each a developer task
  • §7 Open Questions — Unresolved items flagged for human review
The orchestrator verifies all 7 sections exist via Grep before requesting human approval.

Human Approval Required

The design document MUST be approved by a human before implementation begins. The agent presents the design via AskUserQuestion and iterates on feedback. Only after explicit approval does the workflow proceed to Phase 3.

Human Approval Gate

Quality Scoring

6-dimension weighted assessment with automatic fix cycles and a hard 7.0 threshold

Concept
Skill

6 Weighted Dimensions

Quality is measured across six dimensions. Security is weighted highest at 30% because security defects are the most expensive to fix post-deployment.

concept-quality-scoring
Correctness
Does it work as designed?
Security @ 30%
Encryption, IAM, access
Best Practices
Style, conventions, patterns
Testing
Coverage and assertions
Documentation
README, inline comments
Completeness
All checklist items done

Scoring Scale

Each dimension scored 1–10. Weighted average must meet the 7.0 threshold for production readiness. Scores below 7.0 trigger refinement rounds.

tf-judge-criteria

Auto-Fix & Refinement

Unambiguous issues (formatting, missing tags, simple validation errors) are fixed automatically. Ambiguous issues are flagged for review. Maximum 3 refinement rounds before escalation.

The Orchestrator

The main agent that drives the entire SDD workflow from requirements to pull request

Concept

Parses Requirements

The orchestrator receives the initial task from $ARGUMENTS, scans it against the ambiguity taxonomy, and determines what research is needed before design can begin.

concept-orchestrator

Spawns Research Agents

Dispatches 3–4 concurrent research subagents, each investigating one specific question. Research findings are returned in-memory (<500 tokens) to keep the orchestrator's context lean.

Delegates to Specialists

Hands off focused tasks to design agents, test writers, and developer agents. Each subagent reads its own skills (P9 principle) and operates in an isolated context with limited tools.

Trust-but-Verify Gates

After each subagent completes, the orchestrator independently validates the output. It runs terraform validate and terraform test itself, never trusting the subagent's self-reported status.

Handles Human Approval

Presents the design document for human review via AskUserQuestion. Iterates on feedback until explicit approval is given. This is the only blocking point in the workflow.

Human Approval Gate

Produces Final PR

After validation passes (quality score ≥ 7.0), the orchestrator creates the final pull request linked to the tracking GitHub issue. The PR contains all code, tests, and the validation report.

Constitutions

Non-negotiable rules for all code generation

Module Constitution

Non-negotiable rules for Terraform module code generation.

  • File structure: main.tf, variables.tf, outputs.tf, versions.tf, locals.tf, README.md
  • Naming: snake_case everywhere, this for single resource instances
  • Security defaults: encryption enabled, public access blocked
  • tags variable required on all taggable resources
  • No provider blocks in modules
  • Max 500 lines per file
module-constitution

Provider Constitution

Non-negotiable rules for Terraform provider code generation in Go.

  • Go conventions: camelCase locals, PascalCase exports
  • Plugin Framework patterns: types.String, not raw Go types
  • Testing: basic, disappears, fullFeatures, update, validation tests required
  • Error handling: check HasError(), handle NotFound gracefully
provider-constitution

Consumer Constitution

Non-negotiable rules for Terraform consumer (root module) code generation.

  • Module-first composition: raw resources PROHIBITED except glue
  • Dynamic credentials via HCP Terraform
  • Never weaken module defaults
  • Workspace-aware deployment
  • default_tags required: ManagedBy, Environment, Project, Owner
consumer-constitution

Shared Principles

Cross-cutting rules enforced across all three constitutions.

  • Security-first defaults: deny-by-default posture
  • Single design doc as source of truth
  • Quality gates at every phase boundary
  • Deny-by-default for permissions and network access
  • Direct main/master commits prohibited

Authoring Guides

How to create new skills, agents, and constitutions

Writing a Skill

Skills are markdown files that encode reusable domain knowledge for agents.

  • Create .claude/skills/[name]/SKILL.md
  • Add YAML frontmatter: name, description, user-invocable flag
  • Write markdown body with domain knowledge, patterns, and examples
  • Reference via slash command: /[skill-name]
Skills are discovered by name/description matching. Keep descriptions precise so agents select the right skill for each task.

Defining an Agent

Agents are markdown files that define specialized AI assistants with scoped permissions.

  • Create .claude/agents/[name].md
  • YAML frontmatter: name, model, allowedTools, disallowedTools
  • Markdown body contains the system prompt and behavioral instructions
  • Scope tools to the minimum needed for the agent's task
Each agent runs in an isolated context window. Over-permissioning defeats the purpose of multi-agent isolation.

Maintaining Constitutions

Constitutions are non-negotiable rule sets loaded into every agent's context.

  • Located in .foundations/memory/
  • Owned by the platform team
  • Changes require review (PR-based workflow)
  • Must cover: file structure, naming, security, testing
Constitutions are always loaded (P1 priority). They cannot be overridden by skills or user prompts.

Design Template Structure

Templates provide canonical markdown structures that design agents fill in.

  • Located in .foundations/templates/
  • 6-7 mandatory sections per workflow type
  • Shared sections: Security Controls, Implementation Checklist
  • Machine-readable format for downstream agent consumption
Templates ensure completeness and consistency. Downstream agents grep specific sections to extract what they need.
